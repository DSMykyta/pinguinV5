## Система перед кодом

Код підкоряється системі — не навпаки. Якщо нова функція не вписується в шаблон — змінюй код, не систему. Кожен модуль виглядає однаково. Людина знає де шукати і що очікувати ще до того як відкрила файл.

### Три закони

| № | Закон | Опис |
|---|-------|------|
| I | **Один файл — одна роль** | Файл або завантажує плагіни, або зберігає стан, або генерує HTML, або реалізує одну фічу. Ніколи не поєднує. Якщо файл робить дві речі — це два файли. |
| II | **Плагін не знає про інші плагіни** | Єдиний канал між плагінами — `state` через хуки. Плагін не імпортує інший плагін, не викликає функції сусідів. Реагує на події — і все. |
| III | **Система не падає через один плагін** | `Promise.allSettled` — завжди, без виключень. Хуки обгорнуті в `try/catch`. Видаляти плагін безпечно в будь-який момент. |

### Контракт рівнів

Якщо код не відповідає контракту — він лежить не в тому файлі. Переміщуй код, не правила.

| Рівень | Зобов'язаний | Заборонено |
|--------|-------------|-----------|
| `-main.js` | Завантажувати плагіни через `Promise.allSettled`. Експортувати `init*()` або `create*()` | Реалізовувати фічу напряму. Маніпулювати DOM. Містити бізнес-логіку |
| `-state.js` | Зберігати стан. Реалізовувати `registerHook` / `runHook`. Кешувати DOM-посилання | Завантажувати плагіни. Маніпулювати DOM. Містити обробники подій |
| `-template.js` | Генерувати і повертати HTML-рядок на основі конфігурації | Вставляти HTML в DOM. Вішати `addEventListener`. Знати про `state` |
| `-core.js` | Ініціалізувати базову DOM-логіку. Запускати `state.runHook()` в ключових точках | Реалізовувати окремі фічі. Знати про конкретні плагіни |
| `plugin` | Експортувати `function init(state)`. Реагувати через `state.registerHook()`. Завжди передавати `{ plugin: 'name' }`. Якщо є зовнішні підписки — повертати `{ dispose() }` | Імпортувати інші плагіни. Зберігати дані поза `state`. Покладатися на порядок завантаження |

### Мінімальний новий модуль

Три файли — повна ЛЕГО-система. Нова фіча = новий плагін. Видалення фічі = видалення файлу.

```js
// my-main.js — фабрика і завантажувач
import { createMyState } from './my-state.js';

const PLUGINS = [
    './my-feature-a.js',
    './my-feature-b.js',
];

export async function initMyModule() {
    const state = createMyState();
    const results = await Promise.allSettled(PLUGINS.map(p => import(p)));
    results.forEach(r => r.status === 'fulfilled' && r.value.init?.(state));
}
```

```js
// my-state.js — стан і хуки (єдине джерело правди)
export function createMyState() {
    const hooks = {};
    const filters = {};
    return {
        data: [],
        registerHook: (name, fn, opts = {}) => {
            (hooks[name] ??= []).push({ fn, plugin: opts.plugin ?? '?' });
        },
        runHook: (name, ...args) => hooks[name]?.forEach(({ fn, plugin }) => {
            try { fn(...args) } catch (e) { console.error(`[${name}/${plugin}]`, e) }
        }),
        registerFilter: (name, fn) => (filters[name] ??= []).push(fn),
        applyFilter: (name, value, ...args) =>
            (filters[name] ?? []).reduce((v, fn) => fn(v, ...args), value),
    };
}
```

```js
// my-feature-a.js — плагін (можна видалити, система не впаде)
export function init(state) {
    state.registerHook('onDidUpdate', () => { /* реакція на подію */ }, { plugin: 'feature-a' });
    state.registerHook('onWillRender', () => { /* реакція на рендер */ }, { plugin: 'feature-a' });
}
```

### Порушення системи

Якщо бачиш одне з цього — система порушена. Рефактор обов'язковий, виключень немає.

| Порушення | Виправлення |
|-----------|------------|
| Плагін імпортує інший плагін | → Перенести спільну логіку в `-state.js` або `-core.js`. Комунікація тільки через хуки. |
| `-main.js` містить логіку фічі (не просто завантажує плагіни) | → Витягнути у плагін або в `-core.js`. Main — тільки bootstrap. |
| `-state.js` містить `querySelector` або маніпулює DOM | → State зберігає посилання, не маніпулює. DOM-логіка — в `-core.js` або плагін. |
| `-template.js` вставляє HTML в DOM або вішає `addEventListener` | → Template повертає рядок. Вставка — в main, події — в плагін або core. |
| `Promise.all` замість `Promise.allSettled` для завантаження плагінів | → Завжди `allSettled`. Один плагін що впав не зупиняє решту. |
| Плагін зберігає дані в модульній змінній поза `state` | → Весь стан — у `state`. Інакше неможливо ізолювати декілька екземплярів. |
| Плагін покладається на те, що інший плагін вже завантажений | → Плагіни завантажуються паралельно. Порядок не гарантований ніколи. |

## Ядро

Файли без яких модуль не існує. Позначені міткою `🔒 ЯДРО` в шапці. Якщо одного не стає — система не стартує.

### Чотири файли — чотири ролі

Кожен відповідає за одне. Ніколи не поєднуються в один файл.

| Файл | Роль | Опис |
|------|------|------|
| `-main.js` | Фабрика і оркестратор | Створює екземпляр, вставляє HTML в DOM, запускає ядро, завантажує плагіни через `Promise.allSettled`. Єдина точка входу. |
| `-state.js` | Єдине джерело правди | Живий об'єкт що передається в кожен плагін. Зберігає дані, DOM-посилання, реєстр хуків. Плагіни читають і пишуть тільки через нього. |
| `-template.js` | HTML-фабрика | Генерує розмітку компонента на основі конфігурації. Повертає рядок. Нічого не вставляє в DOM, не знає про state. |
| `-core.js` | Базова DOM-логіка | Ініціалізує core-поведінку, вішає базові обробники подій, запускає `state.runHook()` в ключових точках. Не знає про конкретні плагіни. |

### State — спільна шина між усіма рівнями

Плагіни не знають одне про одного. Вони спілкуються виключно через `state` — підписуються на події і запускають їх.

```js
const state = {
    // Дані
    data: [],
    config: { ... },

    // DOM-посилання (кешуються один раз в -main.js)
    dom: { container, toolbar, editor, ... },

    // Хук-шина — реакція на події (side effects)
    registerHook(name, fn, opts = {}) {
        (hooks[name] ??= []).push({ fn, plugin: opts.plugin ?? '?' });
    },
    runHook(name, ...args) {
        hooks[name]?.forEach(({ fn, plugin }) => {
            try { fn(...args) } catch(e) { console.error(`[${name}/${plugin}]`, e) }
        });
    },

    // Фільтр-шина — трансформація даних перед рендером
    registerFilter(name, fn) { (filters[name] ??= []).push(fn); },
    applyFilter(name, value, ...args) {
        return (filters[name] ?? []).reduce((v, fn) => fn(v, ...args), value);
    },
}
```

## Плагіни

Кожен плагін — окремий файл, одна фіча. Завантажується динамічно. Можна видалити будь-який — система продовжує працювати. Позначені `🔌 ПЛАГІН` в шапці.

### Єдиний контракт для всіх плагінів

Один публічний метод. Завжди. Ніяких винятків.

```js
// js/components/editor/editor-formatting.js
// 🔌 ПЛАГІН

export function init(state) {
    const { dom } = state;

    // Підписуємось на події ядра — завжди з { plugin: 'name' }
    state.registerHook('onDidInput', () => updateToolbarState(state), { plugin: 'formatting' });
    state.registerHook('onDidSelectionChange', () => updateToolbarState(state), { plugin: 'formatting' });

    // Реагуємо на UI-події
    dom.toolbar?.addEventListener('click', e => {
        const btn = e.target.closest('[data-action]');
        if (!btn) return;
        applyFormat(state, btn.dataset.action);
    });
}
```

### Як плагіни спілкуються — через хуки

Плагін ніколи не імпортує інший плагін. Тільки підписка і реакція через `state`.

1. **Підписка** (`registerHook`) — При `init(state)` плагін реєструє callback на подію. Може підписатись на кілька хуків.
2. **Тригер** (`runHook`) — Ядро або плагін викликає `state.runHook('onInput')`. Всі підписники отримують виклик паралельно.
3. **Реакція** (`callback`) — Кожен плагін реагує незалежно. Помилка в одному не зупиняє інших — `try/catch` в `runHook`.

### Плагіни в реальних модулях

| Модуль | Плагіни |
|--------|---------|
| `editor/` | formatting, case, undo, validation, find, stats, paste, cleanup, enter, chip-navigation, tooltip |
| `avatar/` | avatar-user, avatar-ui-states, avatar-modal, avatar-selector, avatar-text |
| `table/` | table-sorting, table-filters, table-checkboxes, table-columns |
| Генератори | Кожен generator має власний масив PLUGINS — triggers, reset, copy, aside... |

## Конвенції

Обов'язкові правила оформлення коду. Не пропозиції — стандарт. Кожен новий файл пишеться за цими правилами без виключень.

### Іменування файлів — тип закодований в назві

Відкрив файл в explorer — і вже знаєш що це: ядро, плагін, шарм чи утиліта. Не треба читати код.

| Суфікс / ключове слово | Тип файлу | Що робить | Приклад |
|------------------------|-----------|-----------|---------|
| `-main.js` | Ядро | Оркестратор, точка входу, завантажує плагіни | `editor-main.js`, `products-main.js` |
| `-state.js` | Ядро | Зберігає стан, хуки, фільтри | `editor-state.js`, `products-state.js` |
| `-core.js` | Ядро | Базова DOM-логіка, запускає хуки | `layout-core.js`, `modal-core.js` |
| `-template.js` | Ядро | Генерує HTML-рядок, не вставляє в DOM | `editor-template.js` |
| `-plugin-` | Плагін | Одна фіча, `export function init(state)`, можна видалити | `products-plugin-tabs.js`, `editor-plugin-undo.js` |
| `-charm-` | Шарм | Поведінка через HTML-атрибут, глобальне делегування | `editor-charm-check.js`, `charm-pagination.js` |
| `-data.js` | Дані | API, CRUD операції, парсинг відповідей | `products-data.js`, `brands-data.js` |
| `utils-` | Утиліта | Чиста функція без стану, без DOM. Живе в папці `js/utils/` | `utils-debounce.js`, `utils-json.js`, `utils-file.js` |

**Утиліти** (`js/utils/`) — генеричний код що не належить жодному модулю. Одна роль на файл. Імпортуються будь-ким. Кожен файл має префікс `utils-`, назва описує що файл робить: `utils-debounce.js`, `utils-id.js`, `utils-date.js`.

**Як визначити куди класти код:**

| Питання | Відповідь → Місце |
|---------|------------------|
| Код потрібен тільки одному модулю? | → Плагін цього модуля (`-plugin-`) |
| Код потрібен кільком модулям, працює з DOM? | → Компонент (`js/components/`) |
| Код потрібен кільком модулям, чиста функція? | → Утиліта (`js/utils/`) |
| Код активується HTML-атрибутом? | → Шарм (`-charm-`) |

### Named plugins — ім'я плагіна в кожному хуку

Третій аргумент `registerHook` — завжди об'єкт з іменем плагіна. Коли щось падає — консоль одразу показує де шукати.

```js
// ❌ Без імені — незрозуміла помилка
state.registerHook('onDidUpdate', () => recalc(state));
// Помилка: [onDidUpdate/?] Cannot read properties of null
// → Де шукати? Невідомо.

// ✅ З іменем — помилка вказує на конкретний плагін
state.registerHook('onDidUpdate', () => recalc(state), { plugin: 'stats' });
// Помилка: [onDidUpdate/stats] Cannot read properties of null
// → Одразу знаємо: editor-stats.js
```

### Назви хуків — `onWill` і `onDid`

Префікс говорить чи хук спрацьовує ДО події чи ПІСЛЯ. Читаєш назву — одразу розумієш без коду.

| Префікс | Коли спрацьовує | Приклади |
|---------|----------------|---------|
| `onWill` | Перед подією — можна ще щось підготувати | `onWillRender`, `onWillSave` |
| `onDid` | Після події — вже відбулось, реагуємо на результат | `onDidRender`, `onDidInput`, `onDidSave` |

### Viewless-позначення плагінів без UI

Плагін що не маніпулює DOM позначається `(viewless)` в шапці. Відкрив файл — за секунду зрозумів його роль.

```js
// js/components/editor/editor-stats.js
// 🔌 ПЛАГІН (viewless) — не маніпулює DOM

export function init(state) {
    state.registerHook('onDidInput', () => calcStats(state), { plugin: 'stats' });
}
```

```js
// js/components/editor/editor-formatting.js
// 🔌 ПЛАГІН

export function init(state) {
    // Маніпулює DOM через state.dom.toolbar
}
```

### Disposable — прибирання за собою

Якщо плагін підписується на зовнішні події (`window`, `document`) — він повертає `{ dispose() }`. Core збирає ці об'єкти і викликає `dispose()` при знищенні модуля. Без цього старі callbacks накопичуються в пам'яті при кожній реініціалізації.

```js
// Плагін з зовнішньою підпискою — повертає dispose
export function init(state) {
    const onResize = () => updateLayout(state);
    window.addEventListener('resize', onResize);

    state.registerHook('onDidRender', render, { plugin: 'layout' });

    return {
        dispose() {
            window.removeEventListener('resize', onResize);
        }
    };
}
```

```js
// -main.js — збирає disposables
const results = await Promise.allSettled(PLUGINS.map(p => import(p)));
const disposables = results
    .filter(r => r.status === 'fulfilled')
    .map(r => r.value.init?.(state))
    .filter(Boolean);

// При знищенні модуля:
// disposables.forEach(d => d.dispose?.());
```

### Filter hooks — трансформація даних

Звичайний `runHook` — оголошення події (side effects, реакція). `applyFilter` — плагін отримує значення, трансформує і повертає. Кілька плагінів можуть бути в одному ланцюзі трансформацій не знаючи один про одного.

```js
// editor-null-display.js — замінює null на прочерк
// 🔌 ПЛАГІН (viewless)

export function init(state) {
    state.registerFilter('onRenderCell', (value) => value ?? '—');
}
```

```js
// В core або рендер-плагіні:
const display = state.applyFilter('onRenderCell', rawValue);
// null     → '—'
// 'текст'  → 'текст'
// 0        → 0
```

### -main.js — публічне API модуля

`-main.js` виконує дві ролі одночасно: оркестратор (завантажує плагіни) і публічний API папки (експортує те, що доступне ззовні). Окремий `index.js` не потрібен — у проєкті немає bundler, тому `index.js` не дає жодної переваги і лише створює плутанину у вкладках редактора.

```js
// editor-main.js — оркестратор + публічний API
// Публічний API + оркестратор в одному файлі.
// Зовні імпортуємо тільки звідси.

export async function initEditor(container) { /* ... */ }
export { getEditorState } from './editor-state.js';
```

```js
// ✅ Правильно — через -main.js
import { initEditor } from './components/editor/editor-main.js';

// ❌ Неправильно — прямий доступ до внутрішнього плагіна
import { init } from './components/editor/editor-formatting.js';
```

## Шарми

Готові поведінки що активуються HTML-атрибутом. Нульовий JS на рівні сторінки — достатньо додати атрибут в розмітку.

### Як це працює

Шарми ініціалізуються один раз глобально в `main-core.js` через делегування подій на `document`. Після цього будь-який елемент з потрібним атрибутом отримує поведінку автоматично.

```js
// main-core.js — одна ініціалізація для всіх сторінок
export async function initCore() {
    initSearchClearCharm();   // слухає document, шукає [data-clear-for]
    initFilterPillsCharm();   // слухає document, шукає .filter-pill[data-filter]
    initMorphSearchCharm();   // слухає document, шукає [data-morph-search]
    initRefreshCharm();       // слухає document, шукає [refresh]
    initColumnsCharm();       // слухає document, шукає [columns]
    initPaginationCharm();    // знаходить [pagination] і будує екземпляр
}
```

```html
<!-- HTML на сторінці — тільки атрибут, нуль JS -->

<!-- Кнопка очищення пошуку -->
<input type="text" id="search-brands">
<button data-clear-for="search-brands">×</button>

<!-- Пагінація таблиці -->
<div pagination="25" id="brands-table"></div>

<!-- Кнопка оновлення -->
<div refresh></div>
```

### Довідник шармів

| Шарм | HTML-атрибут | Що робить |
|------|-------------|-----------|
| `charm-search-clear` | `data-clear-for="id"` | Кнопка × для очищення поля пошуку. З'являється коли є текст. |
| `charm-filter-pills` | `.filter-pill[data-filter]` | Чіпи-фільтри — клік перемикає активний фільтр і знімає з інших. |
| `charm-morph-search` | `[data-morph-search]` | Пошук що розгортається з іконки в поле вводу і назад. |
| `charm-refresh` | `refresh` | Кнопка оновлення зі спіннером. Кидає подію `charm:refresh`. |
| `charm-columns` | `columns` | Дропдаун вибору видимих колонок таблиці. |
| `charm-pagination` | `pagination="25"` | Повна пагінація — навігація, розмір сторінки, лічильник. |

## Правила оформлення шапок файлів

Кожен JS файл в проєкті починається з однакової структури: шлях до файлу + блок-коментар з рамкою. Це допомагає зразу зрозуміти що робить файл без потреби читати код.

### Обов'язкова перша рядок

Перший рядок кожного файлу — завжди коментар з реальним шляхом файлу від кореня `js/`.

| Що пишемо | Приклад |
|-----------|---------|
| `// js/шлях/до/файлу.js` | `// js/layout/layout-main.js` |
| Завжди реальний шлях, не старий | Якщо файл переміщено або перейменовано — оновлюємо цей рядок |

### Структура блоку-коментаря

Після першого рядка йде блок-коментар `/* ... */` з рамкою з Unicode символів. Рамка дає чіткий візуальний кордон і легко читається в будь-якому редакторі.

| Елемент рамки | Символи | Призначення |
|---------------|---------|------------|
| Верхня межа | `╔══════╗` | Початок шапки |
| Рядок заголовку | `║ НАЗВА ║` | Назва модуля великими літерами, по центру |
| Розділювач | `╠══════╣` | Відділяє заголовок від опису (якщо є опис) |
| Порожній рядок | `║      ║` | Відступ між блоками тексту |
| Рядок тексту | `║  текст  ║` | Опис, правила, приклади — з відступом 2 пробіли |
| Нижня межа | `╚══════╝` | Кінець шапки |

### Іконки для груп інформації

Всередині блоку використовуються іконки-емодзі щоб позначити тип інформації. Після іконки — список через `├──` і `└──`.

| Іконка | Коли використовувати |
|--------|---------------------|
| 📋 | Перелік правил, можливостей, HTML структури, атрибутів, опцій |
| 🎯 | Приклад використання — як імпортувати і викликати |
| 🔒 | Файл є частиною ядра — не видаляти |
| 🔌 | Файл є плагіном — можна видалити |

### Правила і принципи

| Правило | Пояснення |
|---------|-----------|
| Перший рядок — завжди шлях | Навіть якщо файл переміщено, шлях оновлюємо першим |
| Заголовок — великими літерами | `ПРАВА ПАНЕЛЬ — СТАНИ`, не `Aside State` |
| Розділювач `╠═══╣` — тільки коли є опис | Якщо після заголовку одразу `╚═══╝` — розділювач не потрібен |
| Текст з відступом 2 пробіли від `║` | `║  Текст тут  ║` — два пробіли до тексту |
| Списки через `├──` і `└──` | `├──` — не останній пункт, `└──` — останній |
| Ширина рамки фіксована | 74 символи `═` між кутами — всі рядки мають бути однакової ширини |
| Коротко і по суті | Шапка пояснює що файл робить і як використовувати — не переписуй код словами |
| Не використовувати `*/` всередині коментаря | Символи `*/` всередині блок-коментаря закривають його передчасно |
